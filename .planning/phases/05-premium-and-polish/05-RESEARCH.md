**Researched:** 2026-07-23
**Domain:** Firestore scenario catalog, AI-generated custom scenarios, Today's Twist, Daily Challenge

## Summary

Phase 5 transforms the scenario system from 3 bundled JSONs to a production-scale Firestore-backed catalog, enables unlimited AI-generated custom scenarios, and adds daily engagement features (Today's Twist and Daily Challenge). The app is completely free вҖ” no premium gating, no subscriptions, no payment infrastructure.

The existing codebase already has all the infrastructure needed: Gemini structured JSON output (`EvaluationService` pattern in `ai_service.dart`), Firestore CRUD (`FirestoreService`), SharedPreferences caching (`ConversationStorageService` pattern), and GoRouter navigation with bottom tabs. No new pub packages are required вҖ” all dependencies (`cloud_firestore`, `google_generative_ai`, `shared_preferences`, `flutter_riverpod`, `go_router`) are already in `pubspec.yaml` and resolved in `pubspec.lock`.

**Primary recommendation:** Implement in 4 waves: (1) Firestore-backed catalog + redesigned selection screen, (2) Custom scenario creation flow, (3) Today's Twist badge + progressive variations, (4) Daily Challenge hero card + timezone-aware rotation. Each wave builds on the previous and can be verified independently.

### Pattern 1: Gemini Structured JSON for Scenario Generation
**What:** Use `responseMimeType: 'application/json'` + `responseSchema` to generate structured scenario configs from free-form user input. This is the exact pattern already used by `EvaluationService.evaluateGoal()`.
**When to use:** Scenario generation, twist variation generation, daily challenge generation, and seed prompt generation.

### Pattern 2: Cache-Then-Firestore for Instant Startup
**What:** Load from SharedPreferences cache instantly, then background-fetch from Firestore if cache is stale (>24h). This matches the existing pattern used by `ConversationStorageService`.
**When to use:** For the curated scenarios catalog вҖ” ensures instant UI on repeat visits.
**Where implemented:** `FirestoreScenarioService.getScenarios()` method.

### Pattern 3: FamilyAsyncNotifier for Parameterized ViewModels
**What:** Extends the existing pattern used by `ConversationViewModel` вҖ” `AsyncNotifierProvider.family` scopes state per argument.
**When to use:** ConversationViewModel already uses this pattern with `Scenario` as the family argument. Twist scenarios and Daily Challenge pass through unchanged вҖ” the family argument is the scenario itself.

### Anti-Patterns to Avoid
- **Hardcoding scenario seed data in Dart code:** All curated scenarios should live in Firestore's public `/scenarios` collection. The seed list JSON (`seed_list.json`) is a migration artifact, not a runtime data source.
- **Premium gating in generated scenario plan:** All use cases are unlimited and free per D-01, D-02, D-13. Any "Premium:" comment or gating code is a bug.
- **Client-side-only Daily Challenge date logic without seed document:** Without a Firestore seed document, different clients could see different "today's challenge" due to clock skew. Store the challenge date in Firestore as source of truth.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Gemini structured JSON parsing | Manual JSON schema validation | `responseSchema` in `GenerationConfig` | Already used in `EvaluationService`. Guarantees type-safe output. |
| Firestore CRUD for custom scenarios | Raw Firestore calls in ViewModel | `FirestoreScenarioService` (extend existing) | Existing pattern in `conversation_storage_service.dart`. Testable, injectable. |

**Key insight:** Every infrastructure pattern Phase 5 needs already exists in the codebase. Do not introduce new packages or new architectural patterns. The only genuinely new code is the UI screens (CreateScenarioScreen, Daily Challenge card, search bar) and the Gemini prompt strings.

## Common Pitfalls

### Pitfall 1: Scenario Model Backward Incompatibility
**What goes wrong:** Adding new fields to `Scenario.fromJson()` without defaults causes old cached JSON or Firestore documents to fail deserialization.
**Why it happens:** The existing code saves scenarios as JSON in SharedPreferences and expects `fromJson` to handle all fields.
**How to avoid:** Every new field (tags, difficultyRating, isFeatured, completionCount) MUST use `??` defaults in `fromJson`. Never use `as String` on a potentially missing field.
**Warning signs:** `flutter analyze` passes but runtime `TypeError` occurs on `as String` cast for a null field.

### Pitfall 2: Twist Badge Without Completion Check
**What goes wrong:** Users see Twist badges on scenarios they haven't completed, causing confusion.
**Why it happens:** The Twist badge check only looks at whether a scenario exists in the user's scenario results collection, but doesn't verify it was actually completed (not just started).
**How to avoid:** Check that the scenario has a `bestScore` or a `completedAt` field, not just that a document exists in the scenarios subcollection.
**Warning signs:** Users report "Twist" badges appearing before they've finished a scenario.

### Pitfall 3: Daily Challenge Countdown Timer Memory Leak
**What goes wrong:** The `Timer.periodic` for the 1-minute countdown update is not disposed, causing memory leaks and phantom state updates after navigation.
**Why it happens:** The timer is created in `initState` but not cancelled in `dispose`.
**How to avoid:** Always store the `Timer` in a member variable and call `timer.cancel()` in `dispose()`. For Riverpod StateNotifier, use `ref.onDispose()`.
**Warning signs:** Hot reload cycles accumulate timers; logs show repeated countdown updates after leaving the Home screen.

### Pitfall 4: Custom Scenario ID Collisions
**What goes wrong:** Two custom scenarios from the same user have identical IDs.
**Why it happens:** Using `DateTime.now().millisecondsSinceEpoch` for IDs вҖ” if two scenarios are created rapidly in the same millisecond, IDs collide.
**How to avoid:** Use the `uuid` package (already a dependency) for custom scenario IDs, or append a random suffix to the timestamp-based ID.
**Warning signs:** Rare but produces data loss on second identical ID вҖ” Firestore overwrites silently.

class FirestoreScenarioService {
  final FirebaseFirestore _db = FirebaseFirestore.instance;
  static const String _collection = 'scenarios';
  static const String _cacheKey = 'cached_scenarios';
  static const Duration _cacheTtl = Duration(hours: 24);

  Future<List<Scenario>> getScenarios() async {
    // 1. Load from cache first (instant)
    final cached = await _getCachedScenarios();
    if (cached.isNotEmpty && !_isCacheStale()) {
      return cached;
    }
    // 2. If cache is stale or empty, fetch from Firestore
    final fresh = await _fetchFromFirestore();
    await _cacheScenarios(fresh);
    return fresh;
  }

  Future<List<Scenario>> _fetchFromFirestore({DocumentSnapshot? startAfter, int limit = 20}) async {
    Query query = _db.collection(_collection).orderBy('completionCount', descending: true).limit(limit);
    if (startAfter != null) {
      query = query.startAfterDocument(startAfter);
    }
    final snapshot = await query.get();
    return snapshot.docs.map((doc) => Scenario.fromJson(doc.data() as Map<String, dynamic>)).toList();
  }

  Future<void> _cacheScenarios(List<Scenario> scenarios) async {
    final prefs = await SharedPreferences.getInstance();
    final jsonList = scenarios.map((s) => s.toJson()).toList();
    await prefs.setString(_cacheKey, jsonEncode(jsonList));
    await prefs.setString('${_cacheKey}_timestamp', DateTime.now().toIso8601String());
  }
}

class DailyChallengeService {
  /// Returns the UTC date string (YYYY-MM-DD) for the current challenge day.
  static String get todayChallengeDate => _formatDate(DateTime.now().toUtc());

  /// Returns the UTC date string for the next challenge day.
  static String get nextChallengeDate =>
      _formatDate(DateTime.now().toUtc().add(const Duration(days: 1)));

  /// Calculates seconds remaining until next UTC midnight.
  static Duration get timeUntilNextChallenge {
    final now = DateTime.now().toUtc();
    final nextMidnight = DateTime.utc(now.year, now.month, now.day + 1);
    return nextMidnight.difference(now);
  }

  /// Formats a DateTime to YYYY-MM-DD string in UTC.
  static String _formatDate(DateTime dt) =>
      '${dt.year}-${dt.month.toString().padLeft(2, '0')}-${dt.day.toString().padLeft(2, '0')}';
}

## Open Questions (RESOLVED)

1. **What is the exact Gemini model string used in production?**
   - What we know: `AppConfig.geminiModel` is `'gemini-3.1-flash-lite'` as of the last code snapshot.
   - What's unclear: Whether Gemini 2.0 Flash or the current model better handles structured JSON scenario generation (higher temperature = more creative, but may produce invalid JSON without responseSchema).
   - Recommendation: Use the existing `AppConfig.geminiModel` value. The `responseSchema` constraint ensures valid JSON regardless of model choice.

2. **Should Daily Challenge be globally consistent (all users see the same challenge) or per-user?**
   - What we know: D-06 says "a unique AI-generated variation of a random curated scenario becomes Today's Challenge."
   - What's unclear: "Unique" could mean globally unique (one generated per day, same for everyone) or per-user unique (each user gets a different variation).
   - Recommendation: Globally consistent вҖ” seed one challenge document per day in Firestore at `/challenges/YYYY-MM-DD`. Simpler to implement, consistent experience, supports social discussion ("what's today's challenge?").

3. **Where should twist replay counts be stored?**
   - What we know: D-05 says "Twist is tracked as a visible badge on the card only. No detailed twist history or counter."
   - What we plan: Add a `replayCount` field to the existing `scenario_results` document for each scenario, or store twist-specific data in a `twist_counts` subcollection.
   - Recommendation: Add a `twistReplayCount` field to the existing `users/{uid}/scenarios/{scenarioId}` document. Minimal schema change, integrates with existing data flow.

4. **Should the Daily Challenge seed be generated client-side or via a scheduled Cloud Function?**
   - What we know: D-06 requires daily generation. Cloud Functions cost money; client-side generation is free.
   - Recommendation: Client-side generation is simpler and free. The first user to open the app on a new UTC day triggers the generation and writes to Firestore `/challenges/YYYY-MM-DD`. Subsequent users read the existing document. Rate-limited to one write per day.
