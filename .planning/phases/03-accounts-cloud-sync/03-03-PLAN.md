Add a new method to AuthViewModel: `Future<void> _migrateGuestData(String uid)`:
1. Read SharedPreferences data:
   - `onboarding_language` (String)
   - `onboarding_cefr` (String)
   - `onboarding_goal` (String)
   - `onboarding_completed` (bool)
2. If onboarding_completed is true:
   a. Call `firestoreService.createUserProfile(uid, displayName: "User", email: null)` — create the user doc
   b. Call `firestoreService.savePreferences(uid, language: lang, cefrLevel: cefr, goal: goal)` — save preferences
   c. Call `firestoreService.saveProgress(uid, totalXp: 0, scenariosCompleted: 0)` — initialize progress
3. Clear migrated SharedPreferences keys (set to empty/default) — guest data now lives in Firestore
4. Set `migrationComplete = true` in state

Call `_migrateGuestData` inside `signUpWithEmail()` and `signInWithGoogle()` — but only when the current user was previously anonymous:
- Before calling AuthService.signUp/signInWithGoogle, check `ref.read(currentUserProvider)?.isAnonymous == true`
- After successful auth, if previous user was anonymous, call `_migrateGuestData(newUser.uid)`
- Wrap migration in try/catch: on failure, log error but don't block sign-up. Set `errorMessage` to "Account created but some data could not be synced. Your progress is safe." — local SharedPreferences is NOT cleared on migration failure

For `signInAsGuest()`:
- No migration needed — guest sign-in is the starting point
- After anonymous sign-in, optionally create a Firestore doc for the guest (or skip — guests don't need Firestore until they sign up)

Update `signUpWithEmail()`:
- After successful sign-up, if user was previously anonymous → run migration
- If user was NOT anonymous (fresh sign-up with no prior guest data) → just create empty Firestore profile

Update `signInWithGoogle()`:
- Same logic: check if previous user was anonymous → migrate

The migration is a one-time operation per device. After migration, SharedPreferences guest data is cleared and Firestore is the source of truth.
