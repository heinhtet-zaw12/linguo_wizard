# Research: Phase 3 — Accounts & Cloud Sync

## 1. Current State Analysis

### What Exists (Phase 1+2 Complete)
- **5 features**: splash, onboarding, scenario_selection, conversation, feedback
- **5 services**: ai_service, stt_service, tts_service, evaluation_service, rate_limiter
- **No Firebase dependencies** — app is fully client-side with SharedPreferences
- **No auth** — all data stored locally via SharedPreferences
- **Guest-mode data model**: OnboardingData (language, cefrLevel, goal) stored as simple key-value pairs
- **Navigation**: Named routes in main.dart, simple push/pop

### Data Currently Stored Locally
- `onboarding_completed` (bool)
- `onboarding_language`, `onboarding_cefr_level`, `onboarding_goal` (strings)
- `rate_limit_*` keys (device-based call tracking)
- No user profile, no progress history, no XP persistence beyond session

## 2. Firebase Integration Requirements

### Firebase Auth
- **Email/password** sign up and login
- **Google Sign-In** (via `google_sign_in` package + Firebase Auth)
- **Continue as guest** (anonymous auth or no auth with local-only mode)
- **FlutterFire packages needed**:
  - `firebase_core` — initialization
  - `firebase_auth` — authentication
  - `cloud_firestore` — database
  - `google_sign_in` — Google OAuth

### Firestore Data Model
- Need a clean Firestore user document structure
- Guest-mode data must migrate to authenticated user doc on sign-up
- Current SharedPreferences data needs a migration path

### Proposed Firestore Schema
```
users/{uid}
  ├── profile: { displayName, email, photoUrl, createdAt, lastLoginAt }
  ├── preferences: { language, cefrLevel, goal }
  ├── progress: {
  │     totalXp: number,
  │     scenariosCompleted: number,
  │     lastScenarioAt: timestamp,
  │     streakDays: number,  // Phase 4
  │     streakLastDate: timestamp  // Phase 4
  │   }
  ├── scenarios/{scenarioId}
  │     ├── lastAttempt: timestamp
  │     ├── bestScore: number
  │     ├── attempts: number
  │     └── scores: [{ overall, fluency, grammar, vocabulary, date }]
  └── rateLimits: { dailyCalls: number, date: string }
```

## 3. Home Dashboard Design

### New Feature: `features/home/`
- **HomeScreen** — main landing after onboarding/login
- **HomeViewModel** — manages dashboard state, loads user data
- **Home widgets**: streak ring, daily goal, recommended scenarios

### Navigation Flow Change
```
Current:  SplashScreen → Onboarding → ScenarioSelection → Conversation → Feedback
New:      SplashScreen → Onboarding → Home → ScenarioSelection → Conversation → Feedback

```

### Dashboard Components
1. **Welcome header** — user name or "Guest" with sign-up CTA
2. **Daily goal ring** — circular progress toward daily XP target
3. **Streak indicator** — flame icon + day count (Phase 4 prep)
4. **Recommended scenarios** — 3-4 cards based on CEFR level and history

## 4. Auth Flow Design

### Sign-Up/Login Screens
- **LoginScreen** — email + password fields, Google sign-in button, "Continue as Guest" link
- **SignUpScreen** — email + password + display name, Google sign-up button
- **ForgotPasswordScreen** — email field, reset link

### Auth State Management
- Riverpod `authStateProvider` — stream of `User?` from Firebase Auth
- Route guard: redirect to login if not authenticated and not guest
- Guest mode: no Firebase auth, local-only data (current behavior preserved)

### Guest-to-Authenticated Migration
When a guest signs up:
1. Create Firestore user doc
2. Copy SharedPreferences data → Firestore (onboarding prefs, progress)
3. Optionally: merge local rate limit data
4. Clear local SharedPreferences (now synced to cloud)
5. Update auth state → navigate to Home

## 5. Cloud Sync Strategy

### Sync Approach
- **Online-first with local cache**: Write-through to Firestore, read from Firestore with local cache
- **Conflict resolution**: Last-write-wins (simple for v1)
- **Offline support**: Firestore has built-in offline persistence on mobile

### Data to Sync
- User profile and preferences
- Scenario progress (scores, attempts, XP)
- Rate limiting (per-user, not per-device, once authenticated)

### Data NOT to Sync (Phase 3)
- Conversation transcripts (large, ephemeral)
- STT/TTS audio data
- AI conversation state

## 6. Technical Decisions

### Firebase Configuration
- Need `firebase_options.dart` (generated by `flutterfire configure`)
- Need `.env` updates for Firebase config keys
- Firebase project setup required (Firebase Console)

### Package Additions
```yaml
dependencies:
  firebase_core: ^3.8.0
  firebase_auth: ^5.3.0
  cloud_firestore: ^5.6.0
  google_sign_in: ^6.2.0
```

### Service Layer Changes
- New: `AuthService` — wraps Firebase Auth (signUp, signIn, signInWithGoogle, signOut, guest mode)
- New: `FirestoreService` — wraps Firestore CRUD (user profile, progress, scenarios)
- Modified: `RateLimiterService` — switch from device-based to user-based limits when authenticated
- Modified: `OnboardingViewModel` — save to Firestore when authenticated, SharedPreferences when guest

### Architecture Pattern Compliance
- AuthService and FirestoreService go in `core/services/`
- Auth state management via Riverpod providers in `core/providers/`
- Home feature follows MVVM: `features/home/screens/`, `features/home/viewmodels/`, `features/home/widgets/`
- Auth screens: `features/auth/screens/`, `features/auth/viewmodels/`

## 7. Risks & Considerations

### Firebase Project Setup
- Requires Firebase Console project creation
- Requires `flutterfire configure` for each platform (iOS, Android, Web)
- Requires enabling Auth providers (Email/Password, Google) in Firebase Console

### Data Migration
- Must handle clean migration from SharedPreferences → Firestore
- Must not lose existing user progress during migration
- Consider: what happens if user signs up on device B after using guest mode on device A?

### Platform-Specific
- Google Sign-In requires `GoogleService-Info.plist` (iOS) and `google-services.json` (Android)
- Web requires Firebase JS SDK configuration
- OAuth redirect URIs must be configured

### Security
- Firestore security rules needed (owner-only read/write)
- Auth state must be verified server-side for any future Cloud Functions
- Rate limiting should move to per-user once authenticated

## 8. Implementation Order Recommendation

1. **Firebase setup** — project config, dependencies, initialization
2. **AuthService** — sign up, login, Google sign-in, guest mode, sign out
3. **FirestoreService** — user profile CRUD, progress CRUD
4. **Auth screens** — login, sign up, forgot password
5. **Auth state management** — Riverpod providers, route guards
6. **Home dashboard** — screen, viewmodel, widgets
7. **Guest-to-auth migration** — data copy on sign-up
8. **Navigation update** — add Home route, update flow
9. **Sync existing features** — save progress to Firestore when authenticated
