  <action>
Create `lib/core/services/rate_limiter.dart` as a new service class `RateLimiterService` following the existing service pattern (see AiService as reference). The service must:

1. Use `device_info_plus` to get a stable device identifier:
   - On Android: use `androidInfo.id` (Android ID, stable across OS updates)
   - On iOS: use `iosInfo.identifierForVendor`
   - Fallback: `'unknown-platform'`
   - Do NOT use `androidInfo.fingerprint` — it is not stable across OS updates per Pitfall 4

2. Implement a sliding-window daily counter in SharedPreferences:
   - Key format: `rate_limit_{deviceId}_{YYYY-MM-DD}` (today's date from `DateTime.now().toIso8601String().substring(0, 10)`)
   - This key scheme ensures automatic daily reset without manual cleanup

3. Expose three public methods:
   - `Future<bool> canMakeCall()` — returns true if today's count < maxDailyCalls
   - `Future<void> recordCall()` — increments today's counter atomically (read current, write current + 1)
   - `Future<int> remainingCalls()` — returns maxDailyCalls minus today's count, clamped to [0, maxDailyCalls]

4. Use SharedPreferences fire-and-forget pattern for non-critical writes (per anti-pattern guidance). The `recordCall()` method must await the write to ensure accuracy.

Add to `lib/core/config/app_config.dart`:
- `static const int maxDailyCalls = 10;` (per D-13: 10 calls/day for guest users)
- `static const String rateLimitPrefix = 'rate_limit_';` (key prefix for SharedPreferences)

The class must follow the existing service pattern: plain Dart class, no Flutter widget imports, injectable, no Riverpod dependency in the service itself (Riverpod provider wraps it at call site).
  </action>
